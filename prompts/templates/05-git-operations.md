# 提示词 05：Git 提交流操作

## AI 读取说明

本文件为「AI 自读模式」提示词。用户无需复制文件内容，只需在对话框中输入：

```
使用提示词05：操作：【提交/推送/拉取/状态/日志】 提交信息：【xxx】 行数：【10】
```

示例：
```
使用提示词05：操作：推送 提交信息：更新数据分析页面
```

---

## 前置约束

- **禁止自行执行 Git 操作**：AI 不得在未经用户明确指令的情况下执行任何 Git 命令。所有 Git 操作必须由用户输入「使用提示词05」开头的指令后才能执行。
- **禁止在其他任务中自动提交**：当用户执行文件修改、创建等任务后，AI 不得自动执行 `git add`、`git commit`、`git push` 等操作。必须等待用户明确输入提示词指令。

---

## 执行逻辑

当用户输入「使用提示词05」开头的指令时，AI 需按以下步骤执行：

### 步骤1：解析用户参数

从用户输入中提取以下信息：
- **操作类型**：`提交`、`推送`、`拉取`、`状态`、`日志`（必填）
- **提交信息**：描述本次修改内容，如「添加智能排课功能文档」（提交/推送时必填）
- **行数**：日志显示的行数（仅日志时可选，默认 10）

### 步骤2：操作执行流程

根据操作类型执行对应流程：

#### 2.1 提交（commit）

1. 执行 `git add .` 将所有修改添加到暂存区
2. 执行 `git commit -m "提交信息"` 将暂存区内容提交到本地仓库
3. 告知用户提交成功，显示提交的 hash 和提交信息

**注意**：提交只是将修改保存到本地仓库，不会推送到远程。

#### 2.2 推送（push）

推送前必须先检查本地状态，如有未提交修改先提交，再拉取远程代码防止冲突。执行流程如下：

1. **检查本地状态**：执行 `git status`，检查是否有未提交的本地修改
2. **提交本地修改**（如有）：
   - 如果存在未暂存/未提交的修改 → 执行 `git add .` 和 `git commit -m "{提交信息}"` 先提交到本地
   - 如果没有未提交修改 → 跳过此步骤
3. **拉取远程代码**：执行 `git pull origin {当前分支}`，将远程最新代码拉到本地
4. **检查拉取结果**：
   - 如果显示 "Already up to date" 或无冲突 → 继续执行推送
   - 如果显示冲突 → 停止推送，执行步骤 3 冲突处理
5. **执行推送**：执行 `git push origin {当前分支}`
6. **告知用户推送结果**

#### 2.3 拉取（pull）

拉取前必须先检查本地状态，如有未提交修改先提交，避免拉取时产生冲突。执行流程如下：

1. **检查本地状态**：执行 `git status`，检查是否有未提交的本地修改
2. **提交本地修改**（如有）：
   - 如果存在未暂存/未提交的修改 → 执行 `git add .` 和 `git commit -m "拉取前提交本地修改"` 先提交到本地
   - 如果没有未提交修改 → 跳过此步骤
3. **执行拉取**：执行 `git pull origin {当前分支}`
4. **告知用户拉取结果**
   - 如果无冲突：显示 "拉取成功"
   - 如果有冲突：显示冲突文件列表

#### 2.4 状态（status）

1. 执行 `git status` 查看当前仓库状态
2. 向用户清晰展示：
   - 已修改但未提交的文件（红色）
   - 已提交到本地但未推送的文件（如有）
   - 未跟踪的新文件（如有）

#### 2.5 日志（log）

1. 执行 `git log --oneline -n {行数}` 查看提交历史
2. 向用户展示最近 N 条提交记录，每条显示：
   - 提交的 hash（前 7 位）
   - 提交信息
   - 提交时间（相对时间或绝对时间）

### 步骤3：冲突处理

当执行推送前的拉取操作出现冲突时，AI 必须执行以下步骤：

#### 3.1 识别冲突

1. 执行 `git status` 确认哪些文件有冲突
2. 列出所有冲突文件，向用户说明

#### 3.2 停止推送操作

1. 告知用户冲突已发生，本次推送已取消
2. 明确告诉用户需要手动解决冲突后才能继续

#### 3.3 提供冲突解决指导

向用户提供以下清晰的解决步骤：

**手动解决冲突的步骤：**

1. **打开冲突文件**：使用 VSCode 或任何编辑器打开冲突文件

2. **查找冲突标记**：文件中会有以下标记：
   ```
   <<<<<<< HEAD
   你的修改
   =======
   远程的修改
   >>>>>>> {远程提交 hash}
   ```

3. **选择保留内容**：
   - 只保留「你的修改」部分 → 删除其他内容及冲突标记
   - 只保留「远程的修改」部分 → 删除其他内容及冲突标记
   - 合并两者 → 手动整合内容后删除冲突标记

4. **标记冲突已解决**：所有冲突文件处理完毕后，执行 `git add {冲突文件路径}`

5. **完成解决**：执行 `git commit -m "解决冲突"` 提交解决结果

6. **继续推送**：冲突解决后，重新告诉 AI「使用提示词05：操作：推送 提交信息：解决冲突后重新推送」

#### 3.4 禁止强制操作

**AI 绝对禁止执行以下操作：**
- `git push --force`（强制推送，会覆盖远程历史）
- `git rebase --skip`（跳过冲突，可能丢失代码）
- 直接修改 `.git` 目录下的文件

如遇到复杂冲突无法处理，AI 应告知用户联系有经验的开发者协助。

## 输出要求

### 提交/推送/拉取操作

执行完成后，向用户说明：
1. 执行了哪些 git 命令
2. 命令的输出结果（关键信息）
3. 下一步建议（如有）

### 冲突处理

向用户说明：
1. 检测到哪些文件有冲突
2. 冲突的原因（远程和本地修改了同一处）
3. 具体的手动解决步骤
4. 解决后如何继续推送

### 状态/日志操作

向用户清晰展示命令输出结果，便于用户了解当前仓库状态。

## 参数解析规则

| 参数 | 格式 | 示例 | 必填 |
|------|------|------|------|
| 操作 | 提交 / 推送 / 拉取 / 状态 / 日志 | 推送 | 是 |
| 提交信息 | 任意文本，描述本次修改 | 更新数据分析页面 | 提交/推送时必填 |
| 行数 | 数字 | 10 | 否（日志时可选，默认 10）|

## 示例

### 示例1：提交本地修改

**输入：**
```
使用提示词05：操作：提交 提交信息：添加智能排课功能文档
```

**预期执行：**
1. 执行 `git add .`
2. 执行 `git commit -m "添加智能排课功能文档"`
3. 告知用户提交成功，显示 commit hash

---

### 示例2：推送代码（无冲突）

**输入：**
```
使用提示词05：操作：推送 提交信息：更新数据分析页面
```

**预期执行：**
1. 执行 `git status` → 检查是否有未提交修改（如有则先执行 `git add .` 和 `git commit`）
2. 执行 `git pull origin {当前分支}` → "Already up to date"
3. 执行 `git push origin {当前分支}`
4. 告知用户推送成功

---

### 示例3：推送代码（有冲突）

**输入：**
```
使用提示词05：操作：推送 提交信息：合并新功能
```

**预期执行：**
1. 执行 `git status` → 检查是否有未提交修改（如有则先执行 `git add .` 和 `git commit`）
2. 执行 `git pull origin {当前分支}` → 检测到冲突
3. 停止推送，列出冲突文件：`pages/data-analysis.zh-CN.mdx`、`pages/_meta.zh-CN.json`
4. 向用户说明冲突原因并提供详细解决步骤
5. 告知用户解决后如何继续推送

---

### 示例4：查看仓库状态

**输入：**
```
使用提示词05：操作：状态
```

**预期执行：**
1. 执行 `git status`
2. 向用户展示：
   - 已修改文件列表
   - 未跟踪文件列表
   - 已暂存文件列表（如有）

---

### 示例5：查看提交日志

**输入：**
```
使用提示词05：操作：日志 行数：10
```
